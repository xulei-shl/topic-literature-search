# 代码审查报告 (Code Review Report)
- **审查对象**: `wanfang-search/scripts/wanfang_selection_ops.py`
- **日期**: 2026-05-08
- **总体评分**: 5/10

## 1. 阻断性问题 (Blockers)
> 必须修复，否则“批量全选已成功”这一结论不可靠。

- **位置**: 第 183-205 行、第 288-295 行
- **问题**: 页级全选的“成功校验”依赖 `_extract_selected_count(default_value)`，而调用方把 `default_value` 直接传成 `selected_before_page + expected_increase`。当 `span.checked-tip span.mark-number` 未出现、文本尚未刷新、或解析失败时，函数会直接返回这个“期望值”，导致 `actual_increase` 被误判为已达标。这样会把“未验证”当成“验证通过”。
- **建议**: 将“读取失败”和“数量达标”区分开。读取失败时应返回 `None` 或显式失败，再进入重试/回退流程，不能用期望值冒充真实观测值。

- **位置**: 第 237-256 行、第 313-324 行
- **问题**: JS 兜底直接修改 `input.checked` 和 `span.ivu-checkbox` 的 class，然后 `_is_checkbox_checked()` 优先读取 class。这样即使前端框架状态、已选列表、后端导出上下文都没有同步更新，代码也可能因为 class 被手工加上而误判“已勾选”。对逐条勾选路径来说，`_count_checked_rows()` 和 `_find_unchecked_row_indexes()` 都基于这个判定，存在“界面假选中、逻辑未选中”的风险。
- **建议**: 不要用手工加 class 作为成功标准。应优先验证真实数据源，例如 DOM 已选计数、行级 `input.checked`、或页面上可观测的已选结果区域变化；JS 兜底也应尽量模拟用户点击可交互元素，而不是直接篡改展示 class。

## 2. 优化建议 (Nitpicks)
> 建议修改，提升稳定性。

- **位置**: 第 198-199 行、第 223-225 行
- **问题**: 注释写的是“再次点击确保触发”，但第二次调用 `_ensure_checkbox_checked()` 时，如果复选框样式已经呈现为 checked，函数会直接 `return`，实际上不会再触发点击。也就是说，这个“重试”在很多异常场景下是无效的。
- **建议**: 将“状态检查”和“重试点击”拆开。需要重试时应显式执行一次重新点击，或先取消再点击，并在每一步后做真实状态校验。

- **位置**: 第 41-53 行（`wanfang_page_ops.py`）、第 113-160 行
- **问题**: 当前只等待结果摘要出现，不等待行级 checkbox 元素完成挂载。若列表先渲染、复选框后绑定事件，`checkbox.click()` 失败后又进入 JS 兜底，会放大误判概率。
- **建议**: 在进入勾选流程前，额外等待当前页目标数量的 checkbox 出现，或等待至少首条结果的 checkbox 可见且可交互。
