# 故障复盘报告
- **日期**: 2026-05-08
- **严重级别**: Medium

## 1. 问题现象
高级检索断点续跑时，如果恢复游标已经位于结果集最后一页的末尾位置，例如 `page_text=120/120` 且 `current_row_offset` 已等于当前页记录数，任务虽然实际上已经没有更多可导出的结果，但流程仍会抛出“结果页未选中任何文献，无法导出”，最终将任务状态写成 `failed`。

## 2. 根本原因 (Root Cause)
批次勾选逻辑将“本轮没有选中任何记录”统一视为异常，却没有区分“末页已耗尽”与“勾选失败”这两种语义。因为各站点的 `_select_batch_results` 在检测到无法继续翻页时只是结束循环，所以当本轮 `selected_count` 仍为 `0` 时，后续会直接抛出校验异常；共享导出骨架也缺少对“结果自然耗尽”的显式结束分支，因此把正常结束边界误判成失败。

## 3. 解决方案
在共享 `BatchSelectionResult` 中新增 `reached_end` 标记，并在 `cnki`、`vp`、`wanfang` 三站的批次勾选逻辑中，当游标已到末页末尾且无法继续翻页时返回 `reached_end=True`，不再抛出失败。随后在共享导出骨架 `src/core/advanced_export_flow.py` 中识别该标记：若当前批次 `selected_count=0` 且 `reached_end=True`，则正常结束批次循环，继续执行汇总、进度写回和最终 `success` 收尾流程。

## 4. 预防措施
- 新增共享骨架测试，覆盖“未达到计划导出数但已到结果末尾时仍按 success 收尾”场景。
- 为 `cnki`、`vp`、`wanfang` 分别新增勾选逻辑测试，覆盖“续跑游标位于末页末尾时返回正常结束标记”的边界。
- 后续若新增站点实现高级检索批量导出，应统一复用 `reached_end` 语义，避免再次把自然结束误判为失败。
