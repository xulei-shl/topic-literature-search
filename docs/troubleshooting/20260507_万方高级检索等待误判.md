# 故障复盘报告
- **日期**: 2026-05-07
- **严重级别**: Medium

## 1. 问题现象
执行万方高级检索时，日志显示结束年份已经成功切换到 `2025年`，但点击检索后仍报错“当前页面不是万方搜索结果页”。

同时，部分普通页面点击会出现接近 `PAGE_TIMEOUT` 的长时间停顿，导致一次简单的筛选或切换操作被误认为页面卡死。

## 2. 根本原因 (Root Cause)
因为结果页等待逻辑把 `div.normal-list`、`span.page-number` 这类中间态节点也当成“结果页已就绪”，所以提交后会被过早放行；而结果解析逻辑真正依赖的是 `span.total-number` 结果摘要，所以后续解析阶段又会判定当前不是结果页。

因为万方页面大多是 SPA 交互，但多个点击仍使用 Playwright 默认等待策略，所以像“中英文扩展”、iView 下拉、“检索”按钮这类普通点击也可能被默认导航等待拖到超时。

因为年份下拉原先只记录“已点击目标选项”，没有强制校验 `span.ivu-select-selected-value` 是否真的变成目标年份，所以在页面点击被吞掉、点到错误 dropdown、或组件未完成状态同步时，日志会显示“操作完成”，但界面值仍停留在 `至今`。

## 3. 解决方案
- 将结果页等待收紧为只认结果摘要元素 `span.total-number` / `span.total-number span.mark-number`。
- 将翻页和页码跳转的等待改为围绕目标页码推进，而不是泛化地判断页面状态变化。
- 将下拉、扩展项、检索按钮、每页条数切换统一改为显式动作超时，并关闭默认导航等待，再手动等待目标元素出现。
- 将年份 iView 下拉改为严格验值：若点击后已选值未变成目标年份，则立即重试一次 JS 点击；仍失败时直接抛出异常，而不是继续执行后续检索。

## 4. 预防措施
- 新增测试，确保只有 `div.normal-list` 这类中间态节点时不会误判结果页就绪。
- 新增测试，确保检索按钮和 iView 下拉点击统一使用动作超时与 `no_wait_after`。
- 新增测试，确保严格模式下年份下拉若值未更新会直接报错。
- 后续若继续扩展万方交互，优先遵循“点击只负责动作，等待只负责目标状态”的模式，避免混用默认导航等待。
