# 故障复盘报告
- **日期**: 2026-05-09
- **严重级别**: Medium

## 1. 问题现象
`wanfang-search` 在逐年导出执行到最后一年后，进入年度结果校验阶段时出现 `File is not a zip file`，CLI 将其识别为整轮高级检索失败并不断自动续跑。

典型日志现象：

- 年度子任务进度已写入 `status=success`
- 随后外层流程抛出 `File is not a zip file`
- CLI 连续输出“高级检索执行失败，准备自动续跑”

## 2. 根本原因 (Root Cause)
外层逐年校验会逐年读取每个年度目录中的汇总 `report.txt` 与 `merged.xlsx`。

因为旧实现把“缺少年度报告”“缺少年度合并 Excel”“年度合并 Excel 损坏不可读”都视为致命异常，所以：

- 某个年份的 `merged.xlsx` 不是合法 xlsx 时
- `pandas/openpyxl` 在读取时直接抛出 `File is not a zip file`
- 异常继续向上冒泡到 CLI 自动续跑框架
- CLI 误以为整轮高级检索失败，只能整轮重启
- 由于坏文件仍在，下一轮又会在同一位置失败，形成死循环

## 3. 解决方案
将年度校验从单一“失败即抛异常”改为三类结果：

- `通过`：`report.txt` 总数与 `merged.xlsx` 行数一致
- `重跑`：数量不一致，删除该年子进度文件后重跑该年
- `跳过`：
  - 只有 `YYYY-no-results.txt`
  - 缺少年度汇总报告
  - 缺少年度合并 Excel
  - 年度合并 Excel 无法读取

对 `跳过` 年份：

- 不再触发整轮失败
- 不参与最终总合并
- 在最终总报告中记录“跳过年份”和具体原因

## 4. 预防措施
- 新增 `tests/test_yearly_export_validation.py`，覆盖年度报告与年度结果文件识别
- 新增 `tests/test_wanfang_interactor.py` 场景，覆盖损坏 `merged.xlsx` 时应跳过而不是整轮失败
- 逐年模式 CLI 可读输出新增“跳过年份”，方便在运行结果中直接发现被排除的年度
