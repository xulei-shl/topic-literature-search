# 故障复盘报告
- **日期**: 2026-05-07
- **严重级别**: Medium

## 1. 问题现象
- 执行 `python wanfang-search/scripts/cli.py advanced-search --query "新青年" --date-to 2025 -n 35 --debug` 时，结束年份选择前存在长时间停顿。
- 结果页无法把每页显示条数切换到 `50`。
- 勾选结果后，流程报错 `未能打开万方批量引用页面`，导致 `TXT` 与 `XLS` 导出失败。
- 在实际打开导出页后，日志仍可能报错 `导出标签切换失败: 自定义格式`。
- “中英文扩展”点击后，日志可能停顿接近 `ACTION_TIMEOUT`，随后打印 `关闭中英文扩展失败`。

## 2. 根本原因 (Root Cause)
- 万方结果页的“每页显示条数”组件已从旧的 `ivu-select-item` 结构切换为 `wf-select / wf-option`，但脚本仍按旧选择器定位，导致选项查找失败。
- 年份下拉读取当前值时只兼容 `span.ivu-select-selected-value`。当页面实际渲染为其他文本节点时，Playwright 会在 `inner_text()` 上等待到 `ACTION_TIMEOUT`，形成约 80 秒空档。
- 批量引用页打开逻辑仅通过轮询 `context.pages` 判断新标签，未兼容“当前标签直接渲染导出页”以及新弹窗捕获方式，导致误报打开失败。
- “中英文扩展”关闭逻辑使用 `span.resource-item.active` 作为点击前后的同一个定位器。点击成功后 class 会从 `resource-item active` 变成 `resource-item`，所以后置 `get_attribute()` 会一直等待旧定位器重新出现，最终误判为关闭失败。
- 导出标签切换逻辑点击目标 tab 后，固定读取 `.export-tabs_item` 集合中的第一个节点判断激活态，而不是检查目标 tab 自身状态。页面实际 class token 为 `is-active`，并非单独的 `active`，所以即使导出页已成功切换，旧逻辑仍会持续误判为未激活。

## 3. 解决方案
- 在 `wanfang-search/scripts/wanfang_form_ops.py` 中补充对 `wf-select / wf-option` 的兼容，并把下拉当前值读取改为多选择器兜底，避免无节点时长时间等待。
- 在 `wanfang-search/scripts/wanfang_page_ops.py` 中增加结果页“每页显示条数”文本读取 helper，统一用于等待逻辑。
- 在 `wanfang-search/scripts/wanfang_export_ops.py` 中改为优先使用 `expect_popup` 捕获导出页，失败后兜底识别新标签或当前标签导出页；同时放宽导出标签切换成功判定。
- 在 `wanfang-search/scripts/wanfang_form_ops.py` 中将“中英文扩展”重构为稳定文本节点定位，并在点击后显式等待 `active` 状态消失。
- 在 `wanfang-search/scripts/wanfang_export_ops.py` 中将导出页标签切换重构为“目标 tab 定位 + 目标状态等待”，只检查目标 tab class 与目标导出控件是否就绪。
- 在 `tests/test_wanfang_interactor.py` 中补充“中英文扩展”状态变化测试和“目标 tab 状态判定”测试，覆盖这次误判路径。

## 4. 预防措施
- 针对万方页面中的“选择器易变区域”优先采用“文本标记 + 容器结构”组合定位，减少对单一 class 命名的依赖。
- 新增万方交互测试，覆盖 `cur-text`、`wf-option`、当前标签导出页、无 active class 的导出标签切换、点击后 class 变化的扩展开关等场景。
- 后续站点适配优先先补测试再改选择器，避免页面结构小变更直接回归为运行时故障。
