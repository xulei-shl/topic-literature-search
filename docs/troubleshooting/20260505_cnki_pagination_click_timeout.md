# 故障复盘报告
- **日期**: 2026-05-05
- **严重级别**: Medium

## 1. 问题现象
执行 `python cnki-search/scripts/cli.py advanced-search --query "新青年" --date-to 2025` 等大批量导出任务时，脚本在深页码翻页过程中偶发中断，报错为 `Locator.click: Timeout 30000ms exceeded`，定位到结果页 `#PageNext` 点击阶段。

## 2. 根本原因 (Root Cause)
因为结果页翻页逻辑直接依赖 Playwright 默认 `click()` 等待，所以单次点击会绑定全局 30 秒动作超时；而知网在大批量导出、连续翻到 30 页以后时，前端翻页响应和 DOM 刷新明显变慢，导致异常先在点击阶段抛出，还没进入现有的结果页变化等待逻辑。与此同时，结果页刷新期间的瞬时 DOM 不稳定也会放大超时概率。

## 3. 解决方案
本次修复采用“稳定优先”的最小改动方案：

- 为 `CnkiSearchConfig` 增加 `action_timeout` 与 `page_change_timeout`，拆分单次动作超时和结果页变化超时。
- 将结果页翻页改为“点击发送”与“页面变化等待”解耦，点击 `#PageNext` 时使用 `no_wait_after=True`，避免把翻页成功与 Playwright 内建等待绑死。
- 为 `_goto_next_results_page()` 增加 3 次有限重试，最后一次退回到 JS `click()` 兜底。
- 将结果页全选与逐条勾选改为稳定复选框勾选逻辑，优先走 `check(force=True)`，失败后再退回到 JS 设置 `checked` 并派发事件。
- 为结果页变化等待补充瞬时异常容忍，避免 DOM 刷新中的短暂解析失败直接中断流程。
- 新增单元测试覆盖翻页重试、JS 兜底点击、稳定复选框勾选、页面瞬时刷新容忍。

## 4. 预防措施
- 后续涉及翻页、排序、分页切换等异步页面动作时，默认采用“动作超时”和“页面状态超时”分层设计，不再复用单一全局超时。
- 保留有限重试而非无限等待，避免单次站点抖动直接导致整批任务失败。
- 已新增 `tests/test_cnki_interactor.py` 回归测试，后续调整翻页策略时必须保持这些场景通过。
