# 故障复盘报告
- **日期**: 2026-05-12
- **严重级别**: High

## 1. 问题现象
`vp-search` 在维普结果页执行批量导出时，经常在“每页 50 -> 清空已选 -> 全选/逐条勾选 -> 翻页 -> 循环到 500 条”这条链路上失败。表现包括：

- 明明页面没有真正勾选成功，但本地计数已经到 500，随后导出为空或数量不足。
- 单次“全选 + 翻页”偶尔成功，但进入循环后很快失稳。
- 翻页后程序过早开始勾选，命中了旧 DOM 或半更新状态。

## 2. 根本原因 (Root Cause)
问题不是单一选择器错误，而是三类状态判断同时失真：

1. 当前实现把维普勾选逻辑退化成了“纯 JS 设置 `checked=true` + 本地直接加计数”。因为 Layui 复选框依赖包装层点击和页面事件链更新 UI 计数，所以仅改原生 `input.checked` 并不等于页面真的完成勾选。
2. 批次计数逻辑存在硬伤。因为代码无论勾选是否成功都直接 `page_selected += 1`，所以本地计数会虚高；而最终校验只会在 `UI > 本地` 时修正，不会在 `UI < 本地` 时回退，所以错误会一路带到导出。
3. 翻页完成判定过松。因为之前只要 URL、页码文本或首条标题任意变化就认为翻页成功，而维普是 AJAX 翻页，分页器和局部标题可能先变、结果行后变，所以后续勾选经常发生在旧结果还没彻底替换完的时刻。

## 3. 解决方案
本次修复将维普批量勾选主链路重构为“操作后校验”的模式：

- 恢复“整页全选 -> 数量校验 -> 逐条兜底”链路，不再依赖纯 JS 盲勾选。
- `_ensure_checkbox_checked()` 改为优先原生 `check(force=True)`，失败后再回退到 Layui 包装层点击和 JS 事件兜底。
- `_select_batch_results()` 改为按页统计真实勾选结果，不再无条件累加本地计数，并保留 `reached_end` 正常结束语义。
- `_goto_next_results_page()` 改为要求“目标页码真实推进”才算翻页成功，拒绝“只变标题/只变局部状态”的假成功。
- `每页 50`、`清空已选`、结果页轮询参数统一补强，减少 AJAX 半更新窗口造成的误判。

## 4. 预防措施
- 为维普补量保留并通过以下回归测试：`tests/test_vp_interactor.py`、`tests/test_advanced_export_flow.py`。
- 后续站点如果使用 Layui 或类似包装复选框，禁止把“原生 `checked` 已变”直接等同为“页面业务状态已生效”。
- 结果页翻页等待必须优先使用“页码真实推进 + 结果区稳定”判据，不能只用 URL 或标题变化做成功判定。
