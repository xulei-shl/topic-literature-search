# 故障复盘报告
- **日期**: 2026-05-06
- **严重级别**: Medium

## 1. 问题现象
执行 `python vp-search/scripts/cli.py advanced-search --query "新青年" --date-to 2025 -n 40 --debug` 时，目标导出数量小于每页条数，但第一页逐条勾选后，页面选中数量会直接从 `0` 跳到 `50`，导致本应只选中 `40` 条时误触发整页全选。

## 2. 根本原因 (Root Cause)
结果页复选框采集逻辑兼容了较宽泛的 selector，导致候选集合中可能混入页级全选控件 `selectArticleAll`。因为部分页场景本身会走逐条勾选分支，所以一旦候选集合的首项不是结果行复选框而是页级全选控件，逐条勾选的第一次点击就会把整页记录全部选中。

## 3. 解决方案
在 `vp-search/scripts/interactor.py` 中保留原有“`remaining < 当前页条数` 时走逐条勾选”的主路径，只在 `_current_page_checkbox_items()` 返回前显式排除 `selectArticleAll` 这一类明显的页级全选控件，并在部分页勾选时增加“跳过页级全选，仅逐条勾选”的调试日志。为避免第一页扫描全量结果和逐条读取 `topcount` 导致性能下降，后续进一步对齐 `cnki-search` 的实现节奏：当前页先切片到页内候选，再补齐页内有效结果行；逐条勾选阶段只连续勾选目标区间，不再每勾一条就读取一次已选数量，而是在当前页勾选完成后统一校验一次总数。同时补充并修正 `tests/test_vp_interactor.py` 中相关回归用例，覆盖“混入全选控件时的部分页勾选”“页内连续勾选目标区间”和“过滤后仍保留整页全选优化”三类场景。

## 4. 预防措施
- 为结果页宽泛 selector 增加回归测试，确保候选集合只包含结果行复选框。
- 继续保留整页全选与部分页逐条勾选的分支级测试，避免后续兼容性改动再次混入页级控件。
