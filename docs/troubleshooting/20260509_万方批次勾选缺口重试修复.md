# 故障复盘报告
- **日期**: 2026-05-09
- **严重级别**: High

## 1. 问题现象
万方结果页在批量勾选某一页时，如果页级全选失败后回退逐条勾选，仍然可能出现少量文献未被选中。旧逻辑会把这类“缺口批次”继续当成成功批次导出，导致后续恢复游标继续前移，最终出现漏导。

## 2. 根本原因 (Root Cause)
- 页面勾选逻辑在补勾后仍允许返回不足量结果。因为批次主流程只要求 `selected_count > 0`，所以缺口会被当成部分成功继续推进。
- 批次游标推进依赖实际选中数量和页内偏移共同更新。因为页面已实际向后推进，但当前页存在未勾选项，后续批次会从错误位置继续执行。
- 页面刷新会清空检索条件与结果列表，导致“页内刷新恢复”不可用。批次失败后如果不重建检索上下文，就无法安全回到当前批次起点。

## 3. 解决方案
- 页面勾选补勾后仍未达标时，直接抛出异常，不再允许带缺口继续导出。
- 在共享高级检索骨架中新增批次勾选钩子，由万方站点层接管失败恢复逻辑。
- 整页全选场景不再回退逐条勾选；只要无法确认整页已全部勾上，就立刻放弃当前页并重建检索结果。
- 整页全选前先等待 `wf-checkbox -> label -> span.ivu-checkbox -> span.ivu-checkbox-inner -> input` 结构完整挂载，再按多个候选节点依次尝试点击，降低全选控件局部未绑定导致的误失败。
- 万方批次勾选增加三级恢复链路：
  - 页面勾选失败后先清理已选状态。
  - 优先重新打开高级检索页、回填检索条件、重新提交检索，并恢复到当前批次起点页。
  - 当本地重建次数耗尽后，关闭浏览器并重新启动，再重建当前批次上下文继续重试。

## 4. 预防措施
- 新增单测覆盖“全量导出模式下页面缺口必须直接失败”。
- 新增单测覆盖“当前批次失败后先重建检索上下文再重试”。
- 新增单测覆盖“本地重试耗尽后升级为浏览器级重试”的恢复链路。
