# 故障复盘报告
- **日期**: 2026-05-06
- **严重级别**: Medium

## 1. 问题现象
- 运行 `python vp-search/scripts/cli.py advanced-search --query "新青年" --date-to 2025 -n 133` 时，偶发报错 `未找到"导出题录"入口`。
- 结果页翻页后，执行批量全选或部分逐条勾选前存在明显停顿，部分页场景耗时尤其明显。

## 2. 根本原因 (Root Cause)
- 导出入口识别链路过于依赖短时可见性判断。脚本首次未命中导出入口后，只会点击一次 `批量处理`，再经过很短的等待立即重试；当菜单动画或结果页状态尚未完全稳定时，就会误判为不存在入口。
- 部分页勾选逻辑错误地用“全局已选数量”推断“当前页是否已接近全选”。当前两页已累计选中 100 条、当前页仅 50 条时，会误触发当前页清空逻辑，产生额外等待且无业务收益。

## 3. 解决方案
- 在 `vp-search/scripts/vp_export_ops.py` 中增加“展开批量处理菜单后轮询等待导出入口出现”的逻辑，再执行入口点击；同时保留现有直接点击链路，降低改动范围。
- 在 `vp-search/scripts/vp_search_interactor.py` 中为导出入口补充基于 `导出题录` 文本的选择器兜底，兼容站点 DOM 轻微变化。
- 在 `vp-search/scripts/vp_selection_ops.py` 中移除“部分页目标场景下基于全局已选数清空当前页”的误判逻辑，改为直接逐条勾选目标范围。
- 在 `tests/test_vp_interactor.py` 中补充单测，覆盖“菜单展开后等待导出入口出现”与“部分页勾选不再误清空”的行为。

## 4. 预防措施
- 保留本次新增单测，避免后续修改再次引入入口时序竞态和部分页误清空问题。
- 后续如继续优化翻页后等待时长，应优先围绕“结果列表 DOM 是否真正切换完成”增加更精确的就绪判据，而不是继续叠加固定休眠时间。
