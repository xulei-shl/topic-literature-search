# 故障复盘报告
- **日期**: 2026-05-08
- **严重级别**: Medium

## 1. 问题现象
CNKI 高级检索断点续跑过程中出现两类中断：

1. 首页或高级检索页虽然已经出现可交互元素，但脚本仍因 `Page.goto ... waiting until "load"` 超时而退出。
2. 续跑到导出阶段后，已完成 500 条勾选，但新开的导出页里 `全选` 链接尚未稳定出现，脚本立即报错 `未找到链接: 全选`，并且没有进入自动续跑。

## 2. 根本原因 (Root Cause)
因为 CNKI 页面资源加载较慢，而现有逻辑在恢复会话和打开高级检索页时仍按更严格的页面完成状态等待，所以页面已经具备后续操作条件时，代码还在等待更晚的 `load` 完成，最终触发导航超时。

因为导出页就绪判定只要检测到左侧栏等任一稳定标记就会继续执行，所以代码会过早进入 `全选` 点击逻辑；而 `_click_link_by_text()` 又会在第一次查找不到时直接抛出 `ValidationError`，导致自动续跑逻辑把这类瞬时页面问题误判成非重试错误并直接中断。

## 3. 解决方案
本次修复做了最小化调整：

1. `BrowserManager.restore_session()` 改为使用 `wait_until="domcontentloaded"` 恢复首页和目标页，避免页面可交互后仍死等 `load`。
2. CNKI `_open_advanced_search_page()` 改为优先点击首页 `高级检索` 入口，若点击后页面已满足高级检索特征，则不再强制二次 `goto`。
3. 即使回退到直接打开高级检索 URL，只要当前页面已经满足高级检索特征，也不再因单次导航超时而立即失败。
4. 导出页按文本点击链接逻辑改为轮询等待可见链接，超时时抛出 `TimeoutError`，使其归入可自动续跑的运行时异常。

## 4. 预防措施
- 新增 CNKI 单元测试，覆盖“点击高级检索入口后不再重复 `goto`”场景。
- 新增 CNKI 单元测试，覆盖“导出页链接延迟出现时应重试等待”场景。
- 后续若站点继续存在“元素已可用但页面未完全加载”的情况，应优先收紧到元素级稳定判定，而不是继续依赖 `load` 级导航完成条件。
