# 故障复盘报告
- **日期**: 2026-05-07
- **严重级别**: High

## 1. 问题现象
`vp-search` 在高级检索严格数量导出场景中出现两类问题：

- 勾选阶段明显慢于 `cnki-search` 与 `wanfang-search`
- 已完成目标数量后进入导出阶段，仍可能报错 `等待导出页面打开超时`

在 `python cli.py advanced-search --all --query "新青年" --date-to 2025 -n 175` 场景下，`cnki` 与 `wanfang` 已完成导出，而 `vp` 仍停留在勾选或导出页识别阶段。

## 2. 根本原因 (Root Cause)
`vp-search` 的高级检索导出链路同时存在两类结构性问题：

1. 勾选推进逻辑过重  
   旧实现每一页都依赖 `_extract_selected_count()` 读取全局已选数量，再反推剩余目标数；同时还带有固定稳定等待。这比 `cnki`、`wanfang` 直接按“当前页实际勾选数量”推进的路径更慢。

2. 第一页复选框采集回退过重  
   当第一页分页切片中混入页级全选框时，旧实现会因为过滤后只剩 `49` 条而直接回退到全量可见性扫描，导致单页采集可能耗时二十秒以上。

3. 导出页打开判定不够收敛  
   旧实现点击“导出题录”后，只做统一轮询；并且在 `already_at_target=True` 时会缩短等待窗口。结果“导出页未打开”和“导出页已打开但关键元素未就绪”被混成同一种超时，导致失败定位困难，也更容易误超时。

4. 导出触发路径与页面真实行为不一致  
   页面中的 `li[data-type='abstract']`、`li[data-type='excel']` 只是类型切换标签，真正执行下载的是 `#exportbtn`。旧实现仍保留“点击类型标签后可能自动下载”的分支，导致 `参考文献 -> 导出` 路径不稳定。

## 3. 解决方案
本次重构让 `vp` 的主链路尽量对齐 `cnki`、`wanfang`：

- 勾选阶段改为“页内实勾数量驱动”
  - `_select_rows_on_current_page()` 直接返回当前页实际勾选数量
  - `_select_batch_results()` 按页内实勾数量累计 `selected_count`
  - 全局已选数量读取只保留给页级全选校验和调试兜底

- 第一页分页切片支持顺延补齐结果行
  - 过滤掉页级全选框后，优先补齐到当前页数量
  - 仅在切片仍不可信时才回退到可见性扫描

- 导出阶段改为更明确的打开判定
  - 不再因为 `already_at_target=True` 缩短导出页等待超时
  - 区分“已点击导出题录，但未检测到导出页打开”和“导出页已打开，但关键元素未就绪”
  - 新页与同页切换都通过统一轮询判定

- 下载触发方式统一收敛
  - 先切换导出类型标签
  - 等待目标类型进入 `layui-this`
  - 统一点击 `#exportbtn` 触发下载
  - 若下载结果类型与目标不一致，则强制重切目标标签后重试一次

## 4. 预防措施
- 保留并更新 `tests/test_vp_interactor.py` 中的勾选与导出入口回归测试
- 后续若 `vp` 页面结构再变，应优先补导出入口真实 DOM 样例，再修改选择器集合
- 三库并发场景下，建议持续对齐 `INFO` 级进度日志，避免单库“看起来像卡住”但实际只是日志粒度不一致
