# 故障复盘报告
- **日期**: 2026-05-08
- **严重级别**: High

## 1. 问题现象
万方结果页在执行批量勾选时，页级全选经常判定失败，随后频繁回退为逐条勾选；部分场景下还会出现复选框样式已变化，但真实勾选状态并不可靠的问题。

## 2. 根本原因 (Root Cause)
- 页级全选成功判定依赖已选数量读取，但读取失败时会直接回退到调用方传入的期望值。因为默认值本身就是“预期增加后”的数量，所以会把“未读到真实数量”误判成“校验成功”。
- 万方 `ivu-checkbox` 的 JS 兜底直接修改了 `input.checked` 和样式 class。因为 `_is_checkbox_checked()` 又优先读取样式 class，所以可能把“展示态已变化”误判成“业务态已选中”。
- 结果列表就绪后，复选框本身可能仍在异步挂载。因为勾选逻辑没有等待 checkbox 数量到位，所以页级全选和逐条勾选都可能在控件未完全就绪时触发。

## 3. 解决方案
- 新增 `_read_selected_count()`，将“读取失败”与“读取到具体数量”分离；页级全选不再用默认值冒充真实计数。
- 页级全选优先校验“当前页目标行是否都已真实勾选”，只有在能读到已选数量时才使用数量增量作为辅助判断。
- `_is_checkbox_checked()` 改为优先读取原生 `input` 的勾选状态，只有拿不到 `input` 时才回退读取样式 class。
- `_ensure_checkbox_checked()` 的 JS 兜底改为派发点击事件，不再手工篡改 `ivu-checkbox-checked` 样式。
- 新增 `_wait_for_result_checkboxes()`，在当前页目标复选框数量到位后再进入勾选流程，降低异步渲染导致的误判。

## 4. 预防措施
- 新增单测覆盖“已选数量不可读时，页级全选仍可通过页内真实勾选状态判定成功”。
- 新增单测覆盖“样式 class 与原生 input 状态冲突时，应优先信任 input 状态”。
- 后续如继续优化页级全选，应保持“真实状态优先、展示样式次之”的判定原则，避免再次引入假成功路径。
