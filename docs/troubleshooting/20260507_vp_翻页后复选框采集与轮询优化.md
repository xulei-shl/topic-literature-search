# 故障复盘报告
- **日期**: 2026-05-07
- **严重级别**: Medium

## 1. 问题现象
维普高级检索在结果页翻页后进入下一页勾选判定时，日志显示“结果页翻页完成”之后到“开始清理导出表格”之前存在明显停顿。其中“当前页复选框采集完成”常耗时约 9 秒，第 3 页部分页逐条勾选前的等待体感明显。

## 2. 根本原因 (Root Cause)
翻页后的主要耗时不在固定 `500ms` 稳定等待，而在 `vp-search/scripts/vp_checkbox_list_ops.py` 的当前页复选框采集逻辑。代码在已经按页码切出后续页 `50` 条复选框后，仍对每个元素逐个读取 `name` / `data-name` 属性做结果行判定，导致大量浏览器侧往返调用。

`vp-search/scripts/vp_navigation_ops.py` 的翻页等待循环还会在单轮轮询中多次重复解析结果页摘要，并在部分分支重复读取标题，放大了翻页确认阶段的额外开销。

## 3. 解决方案
对后续页的分页切片路径增加快速通道：当页码大于 `1` 且按页码切片结果已经达到当前页大小时，直接复用切片结果，不再逐项做属性探测。

重构翻页等待循环的单轮逻辑：每轮只解析一次 `parse_results_summary()`，优先使用这份摘要判断页码是否推进；仅在 URL 和页码文本都未变化时，才继续读取首条标题辅助判定状态变化。同时将这份摘要直接返回给调用方，避免翻页成功后再次重复解析。

## 4. 预防措施
新增单元测试覆盖两类回归场景：
- 后续页按页码切片时，不再触发逐项过滤。
- 翻页等待循环单轮只复用一份摘要，且在页码文本已变化时不额外读取标题。

后续若继续优化第 3 页部分页勾选耗时，应单独分析逐条勾选链路，不再与翻页后页面稳定等待混在一起判断。
